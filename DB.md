# База данных

Инструкция для тех, кто разворачивает или администрирует бота (не для повседневной работы с контентом — контентом управляют через `/admin` и `/superuser` прямо в MAX).

## Требования

- PostgreSQL (проверено на 18-й версии, подойдут и более старые — используется только стандартный SQL и `pg_database`).
- Роль (пользователь) с правом **CREATEDB**. У пользователя `postgres` (суперпользователь по умолчанию) это право есть всегда.

## Быстрый старт

1. Скопировать `.env.example` в `.env`.
2. В `.env` указать `DATABASE_URL` — саму базу создавать заранее **не нужно**:

   ```
   DATABASE_URL=postgres://postgres:пароль@localhost:5432/tpgk
   ```

3. Выполнить:

   ```bash
   npm run migrate
   ```

Если базы `tpgk` ещё нет на сервере — миграция создаст её автоматически, затем создаст таблицы, индексы и (только если база пустая) засеет начальный контент колледжа. Если база уже существует — миграция просто приведёт схему к актуальному виду и ничего не удалит и не пересеет.

`npm run migrate` — идемпотентна, её безопасно запускать повторно (например, после обновления кода бота с новыми полями/индексами): существующие таблицы и данные не пересоздаются, только применяются недостающие изменения схемы.

### Как это работает под капотом

`npm run migrate` перед созданием таблиц подключается к служебной базе `postgres` (она есть в любой стандартной установке PostgreSQL) и проверяет, существует ли база из `DATABASE_URL`. Если нет — выполняет `CREATE DATABASE`. Это работает, только если роль в `DATABASE_URL` имеет право `CREATEDB`; если её нет, миграция покажет понятную ошибку с точной командой `CREATE DATABASE ...`, которую нужно выполнить вручную одноразово.

## Схема

| Таблица            | Назначение                                                                 |
| ------------------ | --------------------------------------------------------------------------- |
| `users`             | Роль каждого известного боту человека: `user` / `employee` / `superuser`. Появляется запись только после `/reg` — обычные посетители сайта в таблицу не попадают. |
| `nodes`             | Разделы и подкаталоги главного меню, рекурсивно (`parent_id` ссылается сам на себя). Раздел верхнего уровня — `parent_id IS NULL`. |
| `pages`             | Страницы с текстом, принадлежат конкретному `node`.                          |
| `page_attachments`  | Фото и документы, прикреплённые к странице.                                  |

Полные определения — в [`src/infrastructure/db/migrate.ts`](src/infrastructure/db/migrate.ts).

## Назначение первого суперпользователя

Ролей с самого начала ни у кого нет. Чтобы получить первого `superuser`:

1. Попросите нужного человека один раз отправить боту `/reg` — это создаст его запись в `users` с ролью `user`.
2. Узнайте его `id` (в самой таблице, либо через `/superuser <часть имени>` от лица уже существующего superuser'а, если такой есть) и выполните:

   ```sql
   UPDATE users SET role = 'superuser' WHERE id = <id>;
   ```

Дальше все остальные роли назначаются уже через `/superuser` в самом боте — заходить в базу вручную больше не понадобится.

## Резервное копирование

```bash
pg_dump -U postgres -h localhost -d tpgk --data-only --column-inserts > backup.sql
```

Восстановление — обычный `psql -U postgres -h localhost -d tpgk -f backup.sql` на пустую (только что смигрированную) базу.

## Сброс контента на боевой базе

Если нужно снова полностью очистить контент и засеять заново (см. [`seedContent.ts`](src/infrastructure/db/seedContent.ts)) — таблицы `nodes`/`pages`/`page_attachments` каскадно очищаются одной командой, `users` (роли) она не трогает:

```sql
DELETE FROM nodes;
```

После этого `npm run migrate` увидит, что `nodes` пуста, и заново засеет стартовый контент.
