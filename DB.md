# База данных

Инструкция для тех, кто разворачивает или администрирует бота (не для повседневной работы с контентом — контентом управляют через `/admin` и `/superuser` прямо в MAX, см. [`README.md`](README.md)).

## Требования

- PostgreSQL (проверено на 18-й версии, подойдут и более старые — используется только стандартный SQL и `pg_database`).
- Роль (пользователь) с правом **CREATEDB**. У пользователя `postgres` (суперпользователь по умолчанию) это право есть всегда.

## Быстрый старт

1. Скопировать `.env.example` в `.env`.
2. В `.env` указать `DATABASE_URL` — саму базу создавать заранее **не нужно**:

   ```
   DATABASE_URL=postgres://postgres:пароль@localhost:5432/tpgk-db
   ```

3. Выполнить:

   ```bash
   npm run migrate
   ```

Если базы `tpgk-db` ещё нет на сервере — миграция создаст её автоматически, затем создаст таблицы, индексы и (только если база пустая) засеет начальный контент колледжа. Если база уже существует — миграция просто приведёт схему к актуальному виду и ничего не удалит и не пересеет. Имя базы — любое, `tpgk-db` просто значение по умолчанию в `.env.example`.

`npm run migrate` идемпотентна — её безопасно запускать повторно, например после обновления кода бота с новыми полями/индексами: существующие таблицы и данные не пересоздаются, применяются только недостающие изменения схемы.

### Как это работает под капотом

`npm run migrate` перед созданием таблиц подключается к служебной базе `postgres` (она есть в любой стандартной установке PostgreSQL) и проверяет, существует ли база из `DATABASE_URL`. Если нет — выполняет `CREATE DATABASE`. Это работает, только если роль в `DATABASE_URL` имеет право `CREATEDB`; если её нет, миграция покажет понятную ошибку с точной командой `CREATE DATABASE ...`, которую нужно выполнить вручную одноразово.

## Схема

| Таблица             | Назначение                                                                 |
| -------------------- | --------------------------------------------------------------------------- |
| `users`              | Роль каждого известного боту человека: `user` / `employee` / `superuser`. Появляется запись только после `/reg` — обычные посетители сайта в таблицу не попадают. |
| `nodes`               | Разделы и подкаталоги главного меню, рекурсивно (`parent_id` ссылается сам на себя). Раздел верхнего уровня — `parent_id IS NULL`. |
| `pages`               | Страницы с текстом, принадлежат конкретному `node`.                          |
| `page_attachments`    | Фото и документы, прикреплённые к странице.                                  |

Индексы `nodes(parent_id, sort_order, id)` и `pages(node_id, sort_order, id)` — композитные: покрывают и фильтр, и сортировку сразу, без них выборка страницы списка при большом числе строк требовала бы отдельного шага сортировки. Полные определения — в [`src/infrastructure/db/migrate.ts`](src/infrastructure/db/migrate.ts).

## Роли и назначение первого суперпользователя

Ролей с самого начала ни у кого нет. **Самый первый человек, который когда-либо отправит боту `/reg` на пустой базе, автоматически становится `superuser`** — вручную заходить в БД для этого не нужно.

На практике: сразу после `npm run migrate` на новой базе попросите нужного человека (и только его) отправить `/reg` первым. Все, кто зарегистрируется после — получают обычную роль `user` и дальше повышаются уже через `/superuser` в самом боте (в обе стороны: `user` ↔ `employee` ↔ `superuser`, по одной ступени за раз).

Бот не даст снять права с последнего оставшегося `superuser` — иначе управлять ролями стало бы некому.

Если что-то пошло не так и нужно поправить роль напрямую в БД:

```sql
UPDATE users SET role = 'superuser' WHERE id = <id>;
```

## Резервное копирование

```bash
pg_dump -U postgres -h localhost -d tpgk-db --data-only --column-inserts > backup.sql
```

Восстановление — обычный `psql -U postgres -h localhost -d tpgk-db -f backup.sql` на пустую (только что смигрированную) базу.

## Сброс контента

Если нужно полностью очистить контент (разделы, страницы, вложения) и засеять заново стартовым контентом колледжа (см. [`seedContent.ts`](src/infrastructure/db/seedContent.ts)) — `nodes`/`pages`/`page_attachments` каскадно очищаются одной командой, `users` (роли) она не трогает:

```sql
DELETE FROM nodes;
```

После этого `npm run migrate` увидит, что `nodes` пуста, и заново засеет стартовый контент.
